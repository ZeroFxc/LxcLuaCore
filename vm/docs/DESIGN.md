# Design Document: Pure Lua Register-Based VM and C Translator

## 1. Overview
The project consists of two main components:
1. **A Pure Lua Register-Based Virtual Machine (VM)**: A virtual machine entirely written in Lua 5.1 that can execute a custom register-based instruction set.
2. **A C Translator (Compiler/Transpiler)**: A C program that leverages the official Lua 5.1 C API/source code to parse standard Lua scripts or precompiled Lua 5.1 bytecode and translates it into the custom bytecode format used by the Pure Lua VM.

## 2. Architecture

### 2.1 C Translator
The Translator reads a `.lua` source file or standard Lua 5.1 `.luac` bytecode file. It uses Lua 5.1's parser (via `lua_load` or `luaU_undump`) to generate Lua 5.1 Proto structures (function prototypes). It then iterates over the instructions of these prototypes, maps the standard Lua 5.1 opcodes to our custom opcodes, and writes out a new binary format (the translated bytecode).

**Key Responsibilities:**
- Read standard Lua files.
- Parse/Load into Lua 5.1 AST/Bytecode.
- Translate standard `OP_*` instructions to custom `CUSTOM_OP_*` instructions.
- Serialize the translated functions (constants, instructions, upvalues, nested prototypes) into a custom binary or text format (e.g., JSON or a simple binary format).

### 2.2 Pure Lua Register-Based VM
The Lua VM is written in pure Lua 5.1. It reads the custom bytecode format generated by the Translator and executes it.

**Key Responsibilities:**
- **Loader**: Deserialize the custom bytecode into Lua tables representing functions (Prototypes).
- **State/Execution Context**: Manage the call stack, register windows (similar to Lua's `CallInfo` and `lua_State`), upvalues, and global environment.
- **Dispatcher/Interpreter**: A large loop (or array of functions) that decodes the custom instructions and executes the corresponding operations.

## 3. Instruction Set Design
To differentiate from the standard Lua 5.1 VM, but keep the complexity manageable, we will define a custom register-based instruction set.

### 3.1 Custom Opcodes
We can map standard Lua 5.1 opcodes roughly 1:1 but reorganize them or rename them to make the translation explicit.

Example custom opcodes:
- `OP_MOVE` -> `VOP_MOV` (R(A) := R(B))
- `OP_LOADK` -> `VOP_LDK` (R(A) := Kst(Bx))
- `OP_LOADBOOL` -> `VOP_LDB` (R(A) := (Bool)B; if (C) pc++)
- `OP_LOADNIL` -> `VOP_LDN` (R(A), R(A+1), ..., R(B) := nil)
- `OP_GETUPVAL` -> `VOP_GETUP` (R(A) := UpValue[B])
- `OP_GETGLOBAL` -> `VOP_GETG` (R(A) := Gbl[Kst(Bx)])
- `OP_GETTABLE` -> `VOP_GETT` (R(A) := R(B)[RK(C)])
- `OP_SETGLOBAL` -> `VOP_SETG` (Gbl[Kst(Bx)] := R(A))
- `OP_SETUPVAL` -> `VOP_SETUP` (UpValue[B] := R(A))
- `OP_SETTABLE` -> `VOP_SETT` (R(A)[RK(B)] := RK(C))
- `OP_NEWTABLE` -> `VOP_NEWT` (R(A) := {})
- `OP_SELF` -> `VOP_SELF` (R(A+1) := R(B); R(A) := R(B)[RK(C)])
- `OP_ADD` -> `VOP_ADD` (R(A) := RK(B) + RK(C))
- `OP_SUB` -> `VOP_SUB`
- `OP_MUL` -> `VOP_MUL`
- `OP_DIV` -> `VOP_DIV`
- `OP_MOD` -> `VOP_MOD`
- `OP_POW` -> `VOP_POW`
- `OP_UNM` -> `VOP_UNM` (R(A) := -R(B))
- `OP_NOT` -> `VOP_NOT` (R(A) := not R(B))
- `OP_LEN` -> `VOP_LEN` (R(A) := length of R(B))
- `OP_CONCAT` -> `VOP_CAT` (R(A) := R(B).. ... ..R(C))
- `OP_JMP` -> `VOP_JMP` (pc += sBx)
- `OP_EQ` -> `VOP_EQ` (if ((RK(B) == RK(C)) ~= A) then pc++)
- `OP_LT` -> `VOP_LT`
- `OP_LE` -> `VOP_LE`
- `OP_TEST` -> `VOP_TEST` (if not (R(A) <=> C) then pc++)
- `OP_TESTSET` -> `VOP_TESTS` (if (R(B) <=> C) then R(A) := R(B) else pc++)
- `OP_CALL` -> `VOP_CALL` (R(A), ... ,R(A+C-2) := R(A)(R(A+1), ... ,R(A+B-1)))
- `OP_TAILCALL` -> `VOP_TCALL` (return R(A)(R(A+1), ... ,R(A+B-1)))
- `OP_RETURN` -> `VOP_RET` (return R(A), ... ,R(A+B-2))
- `OP_FORLOOP` -> `VOP_FORL`
- `OP_FORPREP` -> `VOP_FORP`
- `OP_TFORLOOP` -> `VOP_TFORL`
- `OP_SETLIST` -> `VOP_SETL`
- `OP_CLOSE` -> `VOP_CLOSE`
- `OP_CLOSURE` -> `VOP_CLOS` (R(A) := closure(KPROTO[Bx]))
- `OP_VARARG` -> `VOP_VARG` (R(A), R(A+1), ..., R(A+B-2) = vararg)

*(Note: `RK(x)` means Register x or Constant x-256 depending on the 9th bit, identical to Lua 5.1)*

### 3.2 Instruction Encoding
For simplicity in our custom VM, instead of bit-packing instructions into 32-bit integers, the C Translator will output instructions as an array of structures or JSON-like objects. This makes parsing in pure Lua trivial.
For example, an instruction might be stored as:
`{ op = 1 (VOP_MOV), a = 0, b = 1, c = 0 }`

## 4. Custom Bytecode Format
To make the Lua loader simple, the C Translator will generate a JSON-like plain text format, or a simple custom text format, because reading pure binary structs in Lua 5.1 (without `string.unpack`) is cumbersome.

### 4.1 Prototype Structure
A compiled prototype will be represented as:
```lua
{
    numparams = 1,
    is_vararg = 0,
    maxstacksize = 4,
    code = {
        { op = "VOP_GETG", a = 0, bx = 0 },
        { op = "VOP_LDK", a = 1, bx = 1 },
        { op = "VOP_CALL", a = 0, b = 2, c = 1 },
        { op = "VOP_RET", a = 0, b = 1, c = 0 }
    },
    constants = {
        "print",
        "Hello, World!"
    },
    protos = {},
    upvalues = {}
}
```

## 5. Execution Model

### 5.1 VM State
The Lua VM will contain a `state` object:
- `stack`: A flat Lua table acting as the virtual registers.
- `top`: Integer pointer to the top of the stack.
- `callinfo`: A stack of call frames. Each frame contains:
  - `func`: The closure being executed.
  - `pc`: Program counter.
  - `base`: The base index in the `stack` for this frame's registers.
  - `savedpc`: To return to when calling other functions.

### 5.2 Closures and Upvalues
When `VOP_CLOS` is executed, the VM creates a closure object. It captures upvalues from the current frame.
Upvalues will be implemented as tables `{ value = x, closed = false }`. If closed, the value is stored in the table. If open, it points to a stack index.

### 5.3 Dispatch Loop
The core of the VM is a loop:
```lua
while true do
    local inst = code[pc]
    pc = pc + 1
    local op = inst.op
    if op == "VOP_MOV" then
        stack[base + inst.a] = stack[base + inst.b]
    elseif op == "VOP_ADD" then
        -- Handle RK logic
        -- Perform addition
    -- ...
    end
end
```

## 6. Development Stages
1. **C Translator Core**: Integrate Lua 5.1 parser, extract prototypes, and build JSON/Text exporter.
2. **Lua VM Core**: Implement the basic state, stack, and loader.
3. **VM Instructions (Math/Logic)**: Implement arithmetic, logic, and move instructions.
4. **VM Instructions (Functions/Tables)**: Implement calls, returns, closures, and table operations.
5. **Testing**: Run a suite of Lua tests through the Translator and execute them on the pure Lua VM.
